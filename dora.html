Meta | 2021

Exploring mobile AR navigation to allow Meta employees to test new technologies

While at Meta, I worked on several apps that were intended to allow staff to test emerging technologies. Although these products didn't possess the challenges that come with launching something publicly, they did come with their own unique problems as well as opportunities to try different solutions.

The goal of one app was to test out a new technology that uses 3D maps to provide hyper-accurate positioning. This technology is critical to the expansion of the Metaverse in ensuring AR assets can be positioned accurately within the real world. By creating an internal product, insights could be gathered on the current performance of the technology, as well as explore a potential future application of it.

Mapping the flow

Being an internal tool utilizing an emerging technology, the core of the design was not around the user needs, but instead around the state of tech. As such, the first challenge was to work with engineers to understand how the technology should work, and more importantly, what would happen when it didn't.

This collaboration allowed me to first define the states that could occur with the technology, and then map out these in isolation of any UX.

  • Available: 3D maps should be available based on GPS position
  • Unavailable: 3D maps not be available based on GPS position
  • (Re)localzing: the act of the device positioning itself within the 3D map
  • Active*: device positioned
  • Error: positioning lost (but retrying)
  • Failed: positioning lost and attempts to retry failed

Having established this and corroborated with the engineering teams, I was able to start mapping out the greater UX flow around it. This allowed us to start identifying areas that needed further investigation.

Challenges of a global pandemic

Working on this project during a global pandemic and not being able to meet up to be able to discuss the details of the technology added an extra challenge to overcome. As the tech was still being developed, we frequently found ourselves getting lost in abstract concepts.

To overcome this, I built a simulator using Mapbox and Mapillary JS to allow the team to remotely walk down a street together. This tool became central to our discussions and allowed us to align on the definitions above.

UX/UI Challenges

Having mapped out the core flow, a key UX/UI challenge was identified: the user would likely switch between being camera-based to map-based throughout a journey (eg if the tech fails / a journey spans mapped and non-mapped areas).

We had also already identified the user need of being able to choose which navigation solution they utilized (even though this was an internal tool for testing, we wanted to make it as usable as possible to maximize insights as well as help the company's understanding of mobile AR design).

While exploring potential solutions, the concept of an adaptive combined UI emerged that allowed for a seamless transition between the navigation methods. This concept was borne out of the observation that when a user is navigating using AR, they are holding the phone up; when navigating with a map, they are likely to hold the phone down.

As this was a fairly novel solution that utilized intuitive movement, Figma mockups weren't enough to fully explore the concept. To determine whether an adaptive UI based on phone position would be intuitive or be confusing, I turned to javascript to be able to prototype this solution effectively. This allowed me to refine the concept, as well as allow the team to test out the UI for themselves (it used real data, so everyone could try it out themselves at their own location).

Although this took a little extra time, the benefits outweighed the costs. Giving the key stakeholders the ability to test it out themselves provided collective confidence in the solution, and that the extra implementation effort was worth it. With this, I set out fleshing out the full UX/UI with high fidelity mockups.

Final UI

With the prototype tested, I then refined the UI for the engineers to build. Whilst this was only an internal app, we wanted to make it feel as 'finished' as possible to encourage people to use it.

Although some aspects required bespoke design (eg to accommodate the custom interaction, and the loading iconography), existing design system components were utilized as much as possible. This helped reduce the engineering effort and opened up the possibility of any new patterns to be adopted by other teams.

Conclusion

The work here not only allowed the engineering teams to gather data on the performance of their technology, but also added insights to the potential design solutions for future mobile AR apps.
